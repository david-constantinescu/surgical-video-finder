# Internationalization (English / Romanian)

## UI strings

Translations live in:

- [`app/i18n/en.json`](../app/i18n/en.json)
- [`app/i18n/ro.json`](../app/i18n/ro.json)

The frontend loads the active bundle from `GET /api/i18n/{lang}` and applies strings via `data-i18n` attributes.

## Switching language

- Header toggle **EN** / **RO**
- Persisted in `localStorage` key `svf_lang`
- Passed to all API calls as `?lang=ro`

## Terminology

| Layer | English | Romanian |
|-------|---------|----------|
| Curated NLM | `term_translations` + synonyms | Glossary + overrides CSV |
| ICD-10-CM | CDC import | Crosswalk when icd10.ro Excel available |
| ICD-10-PCS | CMS import | — |
| ACHI procedures | — | Parsed from SANT PDF |

### Fallback behavior

If `term_translations.locale = 'ro'` is missing:

- UI shows English `primary_name` with `[EN]` badge (`fallback_en: true` in API).
- Semantic search still works: embeddings include both languages when available.

### Editing Romanian labels

1. Add rows to [`data/curated_ro_overrides.csv`](../data/curated_ro_overrides.csv) (`term_id` or English name → Romanian label).
2. Re-run `python scripts/translate_curated_ro.py`.

## Video search queries

`app/video_links.py` uses Romanian display names when `lang=ro` and appends:

- `chirurgie`, `operatie`, `video operatie` for RO YouTube / hospital sources
- `surgery`, `operative video` for English sources

## Search tokenizer

FTS5 uses `unicode61` tokenizer to support Romanian diacritics (ă, â, î, ș, ț).

## Adding a new language

1. Add locale to `SUPPORTED_LOCALES` in `app/config.py`.
2. Create `app/i18n/{locale}.json`.
3. Extend `term_translations` CHECK constraint and import pipeline.
4. Use a multilingual embedding model (already `paraphrase-multilingual-MiniLM-L12-v2`).
